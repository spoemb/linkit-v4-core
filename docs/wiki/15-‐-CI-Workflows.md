# CI Workflows

This page documents the GitHub Actions workflows that build firmware images and run the host test suite. Both live in [`.github/workflows/`](https://github.com/arribada/linkit-v4-core/tree/main/.github/workflows) of the main repo.

| Workflow | File | Trigger | Purpose |
|----------|------|---------|---------|
| **Build Firmware** | `build.yml` | Manual (`workflow_dispatch`) | Compile a fully customized firmware variant and download the artifacts |
| **Tests** | `tests.yml` | `push` / `pull_request` to `main`, manual | Run unit tests, SWS tests and the turtle simulation |

---

## 1. Build Firmware (`build.yml`)

Manual-only workflow. The user picks a config from the Actions UI ("Run workflow") and downloads the produced firmware. Replaces the legacy `Jenkinsfile`.

### How to run

1. Open **Actions** → **Build Firmware** → **Run workflow** (top right).
2. Fill the form (every input is documented inline):
   - **Board** (`LINKIT` / `RSPB`)
   - **Comm module** (`KIM` / `SMD_SPI` / `SMD_UART` / `LORA`)
   - **Sensors** — independent checkboxes for `AXL`, `PRESSURE`, `THERMISTOR`, `SEA_TEMP`, `ALS`, `CDT`, `PH`, `MORTALITY`, `SWS_LOG`, `GNSS_BBR`, `CAM`
   - **System features** — `BUZZER`, `BATTERY_MONITOR_TYPE`, `LORA_DCS`, `LORA_POWER_OFF_UNDERWATER`, `DEBUG_LEVEL`
3. Click **Run workflow**. ~5 min on the first run (Docker image cold-build), ~30 s once the layer cache is warm.
4. Open the run, scroll to **Artifacts**, click the zip to download.

### Inputs

#### Build mode

| Input | Type | Default | Notes |
|-------|------|---------|-------|
| `board` | choice | `LINKIT` | `LINKIT` or `RSPB` |
| `comm` | choice | `KIM` | `KIM`, `SMD_SPI`, `SMD_UART`, `LORA`. `LORA` requires `board=LINKIT` |

#### Sensors

| Input | Default | CMake flag |
|-------|---------|------------|
| `sensor_axl` | true | `ENABLE_AXL_SENSOR` |
| `sensor_pressure` | false | `ENABLE_PRESSURE_SENSOR` |
| `sensor_thermistor` | false | `ENABLE_THERMISTOR_SENSOR` — **exclusive with `sea_temp`** |
| `sensor_sea_temp` | false | `ENABLE_SEA_TEMP_SENSOR` — **exclusive with `thermistor`** |
| `sensor_als` | false | `ENABLE_ALS_SENSOR` |
| `sensor_cdt` | false | `ENABLE_CDT_SENSOR` |
| `sensor_ph` | false | `ENABLE_PH_SENSOR` |
| `sensor_mortality` | false | `ENABLE_MORTALITY_SENSOR` — auto-enables `axl` + `thermistor` |
| `sensor_sws_log` | false | `ENABLE_SWS_LOG` |
| `sensor_gnss_bbr` | false | `GNSS_HAS_BACKUP_BATTERY` |
| `sensor_cam` | false | `CAM_ENABLE` |

#### System features

| Input | Default | CMake flag |
|-------|---------|------------|
| `buzzer` | false | `BUZZER_ENABLE` |
| `battery_monitor` | `ANALOG` | `BATTERY_MONITOR_TYPE` (`ANALOG` / `STC3117` / `FAKE`) |
| `lora_dcs` | true | `LORA_DCS_ENABLE` (LoRa duty-cycle ETSI EN 300 220) |
| `lora_power_off_underwater` | false | `LORA_POWER_OFF_UNDERWATER` |
| `debug_level` | `3` | `DEBUG_LEVEL` (0=off … 4=trace) |

### Validation

The `Validate inputs and resolve config` step rejects invalid combinations before launching cmake:
- `comm=LORA` with `board=RSPB` → **error**
- `sensor_thermistor` and `sensor_sea_temp` both true → **error**
- `sensor_mortality=true` with `sea_temp=true` → **error** (mortality needs thermistor, which is mutually exclusive with sea_temp)
- `sensor_mortality=true` with `axl=false` or `thermistor=false` → **warning** + auto-enable

### Required secret

Create a repository secret to enable signing of the merged firmware image:

| Secret | Value |
|--------|-------|
| `NRFUTIL_PKG_KEY_PEM` | Full PEM contents of `ports/nrf52840/nrfutil_pkg_key.pem`, including the `-----BEGIN…-----` and `-----END…-----` lines |

**Path**: Settings → Secrets and variables → Actions → New repository secret.

If absent: the workflow still runs and produces `*.hex` / `*.elf` / `*_dfu.zip`, but skips `*_merged.hex` and `*_app_settings.hex` — flashing a blank board with `nrfjprog` won't be possible without the merged image.

### Artifacts

The artifact bundle is named `<variant>-<TAG>` (e.g. `linkit-smd-uart-v4.1.1-6-g5fb1c383`) and contains:

| File | Purpose |
|------|---------|
| `LinkIt_board-<TAG>.hex` *(or `LinkIt_RSPB_board-…`)* | Application only — `nrfjprog --program ... --sectorerase` (bootloader + SD must already be flashed) |
| `LinkIt_board_merged-<TAG>.hex` | Application + bootloader + softdevice + settings — `nrfjprog --program ... --chiperase` for a blank board |
| `LinkIt_board_app_settings-<TAG>.hex` | App + settings — `nrfjprog --program ... --sectorerase --verify --reset` |
| `LinkIt_board_dfu-<TAG>.zip` | Signed DFU OTA package (BLE / serial DFU) |
| `LinkIt_board_dfu-<TAG>.bin` | Raw binary of the application hex |
| `LinkIt_board-<TAG>.elf` | Symbol file for GDB / Ozone |
| `LinkIt_board-<TAG>.img` | Internal nRF image (when generated by the build) |
| `build_config.txt` | Full record of every CMake flag used, plus commit hash and date — keep it next to the binaries for traceability |

### Variant directory mapping

The CMake build dir is variant-specific so multiple builds don't collide:

| `board` × `comm` | Build dir |
|------------------|-----------|
| `LINKIT` × `KIM` | `ports/nrf52840/build/LINKIT` |
| `LINKIT` × `SMD_SPI` | `ports/nrf52840/build/LINKIT_SMD_SPI` |
| `LINKIT` × `SMD_UART` | `ports/nrf52840/build/LINKIT_SMD_UART` |
| `LINKIT` × `LORA` | `ports/nrf52840/build/LINKIT_LORA` |
| `RSPB` × `KIM` | `ports/nrf52840/build/RSPB_KIM` |
| `RSPB` × `SMD_SPI` | `ports/nrf52840/build/RSPB_SMD_SPI` |
| `RSPB` × `SMD_UART` | `ports/nrf52840/build/RSPB_SMD_UART` |

### Build environment

The workflow reuses the project [Dockerfile](https://github.com/arribada/linkit-v4-core/blob/main/Dockerfile) (Ubuntu 22.04 + ARM GCC 10.3-2021.10 + nrf-command-line-tools + nrfutil + mergehex). The image is built with `docker/build-push-action@v5` using GitHub Actions layer cache (`type=gha`), so subsequent runs reuse the previous image layers instead of rebuilding from scratch.

## 2. Tests (`tests.yml`)

Runs on every push to `main`, every pull request to `main`, and on manual dispatch. Three test jobs run in parallel after a shared compile step.

### Jobs

| Job | Command | Notes |
|-----|---------|-------|
| **Build host test binaries** | `cmake -GNinja .. && ninja TrackerTests TurtleSimulation` | Native Linux build (no ARM toolchain). Binaries shared as a 1-day artifact |
| **Unit tests** | `./TrackerTests -ojunit -p -xg SWS` | All groups except SWS. `-p` isolates each test in its own subprocess |
| **SWS tests** | `./TrackerTests -p -g SWSAnalog` *and* `-g SWSAnalogFlash` | Long-running underwater simulation (`TEST_GROUP(SWSAnalog)` and `TEST_GROUP(SWSAnalogFlash)`) |
| **Turtle simulation** | `./TurtleSimulation -v` | 1-year mission simulation. Sets `TURTLE_REPORT_PATH` so the binary emits an HTML report |

`concurrency` cancels in-progress runs on the same ref so a new push supersedes the previous one.

### Artifacts

| Artifact | Content |
|----------|---------|
| `test-binaries` | `TrackerTests` + `TurtleSimulation` (1-day retention, internal) |
| `unit-test-reports` | JUnit XML per group |
| `sws-test-reports` | JUnit XML for `SWSAnalog` + `SWSAnalogFlash` |
| `turtle-simulation-report` | `turtle_simulation_<run_id>.html` + `turtle_simulation_latest.html` (30-day retention) |

JUnit reports are produced via CppUTest's `-ojunit` flag and stored under `tests/reports/{unit,sws}/`.

### Adding a new test group

1. Add `mything_test.cpp` under [`tests/src/`](https://github.com/arribada/linkit-v4-core/tree/main/tests/src) using `TEST_GROUP(MyGroup)`.
2. Register the source in [`tests/CMakeLists.txt`](https://github.com/arribada/linkit-v4-core/blob/main/tests/CMakeLists.txt) under `target_sources(${PROJECT_NAME} ...)`.
3. The unit-tests job will pick it up automatically (it runs everything except `SWS`). If your group is intentionally long-running, exclude it via `-xg <YourGroup>` and run it in its own job.

## Maintenance notes

- **Updating the Dockerfile** (e.g. new ARM GCC version): edit [`Dockerfile`](https://github.com/arribada/linkit-v4-core/blob/main/Dockerfile) at the repo root. The next workflow run rebuilds the image once and re-caches it.
- **Action versions**: GitHub will deprecate Node 20 actions on Sept 16th, 2026. Bump `actions/checkout`, `actions/upload-artifact`, `actions/download-artifact`, `docker/setup-buildx-action`, `docker/build-push-action` when newer Node 24 versions are released.
- **Workflow input limit**: GitHub allows up to **25 inputs** per `workflow_dispatch`. The current `build.yml` uses 18, so there is room for extra flags.
- **Removed**: the legacy `Jenkinsfile` was deleted when these workflows landed. If a private fork still references it, switch to the GitHub Actions workflows.
